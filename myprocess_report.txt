# Report: Iterative Development and Troubleshooting of a RAG-based Microsoft Teams Support Assistant

## 1. Introduction

This report documents the complete development journey of my Retrieval-Augmented Generation (RAG) chatbot project for Microsoft Teams troubleshooting. Instead of a straight-line implementation, this project evolved through multiple iterations, debugging phases, and design corrections. Each correction was driven by observed failures (incomplete answers, repetition, truncation, poor retrieval) and followed by targeted fixes.

The purpose of this report is to demonstrate **problem-solving ability, system-level thinking, and understanding of RAG trade-offs**, rather than just presenting a final working system.

---

## 2. Initial Problem Statement

The goal was to build a chatbot that:

* Answers Microsoft Teams troubleshooting queries
* Uses *only* provided documentation (no hallucination)
* Works offline for retrieval
* Produces step-by-step, actionable solutions

The initial dataset consisted of a **large PDF (~890 pages)** of Teams troubleshooting documentation.

---

## 3. Data Extraction Challenges

### 3.1 Raw PDF Issues

Problems observed:

* Headers and footers repeated on every page
* Page numbers, dates, and "Applies to" sections polluting content
* Commands and steps breaking across page boundaries

### 3.2 Correction Applied

* Counted line frequency across pages
* Removed lines exceeding a repetition threshold (headers/footers)
* Preserved step continuity instead of deleting aggressively

**Key learning:** Over-cleaning removes important troubleshooting steps. Cleaning must be *conservative*.

---

## 4. Issue Detection and Deduplication

### 4.1 Initial Issue Detection

Issue titles were detected using regex patterns ("resolve", "issues", "not launching", etc.).

### 4.2 Problems Found

* Many duplicate issue titles
* Some titles were procedural instructions rather than real issues
* Some "bad titles" actually contained important steps

### 4.3 Corrections

* Introduced `is_bad_title()` to prevent procedural lines from becoming issue boundaries
* Ensured bad titles were **retained as content**, not deleted
* Normalized issue titles (removing dates, "Applies to", repeated suffixes)

**Result:** Reduced from hundreds of noisy issues to **~19–25 clean, unique issues**.

---

## 5. Chunking Strategy Evolution

### 5.1 Initial Chunking (Too Large)

* Initial chunks exceeded 600–1000 words
* Gemini/OpenAI responses were truncated
* FAISS retrieval returned overlapping content

### 5.2 Corrections Made

* Introduced chunk size limits
* Split content into **~250–300 word chunks**
* Enforced minimum chunk size to remove tiny fragments

Example correction:

```python
final_chunks = [
    c for c in cleaned_chunks
    if len(c["text"].split()) >= MIN_WORDS
]
```

**Result:** ~64–120 high-quality chunks depending on filtering.

---

## 6. Embedding and FAISS Indexing

### 6.1 Initial Setup

* SentenceTransformer: `all-MiniLM-L6-v2`
* FAISS: `IndexFlatIP` with normalized embeddings (cosine similarity)

### 6.2 Observations

* FAISS retrieval worked correctly
* Problems were not in embeddings but in *context assembly*

---

## 7. Retrieval Failures and Fixes

### 7.1 Problem: Repetition & Restarting Steps

Symptoms:

* "Clear Teams Cache" repeated multiple times
* Answers restarting mid-way
* Output truncating even with high token limits

### 7.2 Root Cause Identified

* Multiple chunks from the **same issue** containing overlapping headings
* Model confusion when forced into bullet-point formatting

### 7.3 Corrections Applied

* Retrieved more chunks (`k=6`) but passed only **top 3 chunks** to the LLM
* Restricted generation to **one dominant issue** using frequency counting
* Deduplicated chunks by text, not just title

```python
filtered_chunks = filtered_chunks[:3]
filtered_chunks = dedupe_by_text(filtered_chunks)
```

**Key learning:** More context ≠ better answers. Coherence matters more than quantity.

---

## 8. Prompt Engineering Iterations

### 8.1 Initial Prompt (Too Strict)

* Model frequently replied: "I couldn't find a documented solution"

### 8.2 Adjustments

* Allowed combining *partial steps* within the same method
* Prevented repeating section headers
* Controlled formatting instructions

Final stabilized instruction:

```
Provide clear, step-by-step instructions. Use numbered steps where appropriate.
```

---

## 9. Token Budget Debugging

### 9.1 Observed Issue

* 279-word chunks caused truncation despite `max_output_tokens=1600`

### 9.2 Analysis

* RAG token usage includes:

  * System prompt
  * Retrieved context
  * User query
  * Generated answer

### 9.3 Resolution

* Reduced number of chunks passed to LLM
* Set `max_output_tokens ≈ 1200–1500`
* Implemented "continue if cut off" safeguard

---

## 10. Model Selection and API Issues

### 10.1 OpenAI API

* API quota error despite ChatGPT Go plan
* Learned distinction between ChatGPT subscription and API billing

### 10.2 Switch to Gemini API

* Used Gemini free tier
* Achieved stable generation
* Maintained grounding guarantees

**Key learning:** RAG pipelines should be LLM-agnostic.

---

## 11. UI Integration

* Built Streamlit UI for interaction
* Markdown rendering solved formatting issues
* Added caching for FAISS and embeddings

---

## 12. Architectural Decisions

### 12.1 Why Not PostgreSQL

* Static, read-only dataset
* No meaningful relational queries
* JSON + FAISS sufficient

Postgres reserved as a **future extension** for metadata filtering or multi-service scaling.

---

## 13. Final Outcome

* Clean, grounded troubleshooting answers
* No hallucination
* Stable formatting
* Internship-ready architecture

---

## 14. Key Takeaways

* RAG failures are usually *context and structure problems*, not model problems
* Deduplication and chunk selection are more important than model size
* Less context often produces better answers
* Iterative debugging is essential for real-world RAG systems

---

## 15. Conclusion

This project demonstrates not only the ability to build a RAG system, but also the ability to **debug, reason about, and improve it systematically**. The final system is the result of multiple informed corrections rather than trial-and-error, reflecting real-world engineering practice.
